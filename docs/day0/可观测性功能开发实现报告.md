`TokenUsageTracker.track()` 方法已经完美实现，包括 Counter/Timer 注册和会话统计，但 **从未被调用**。

**根因**: `ModelFallbackService.callWithRoles()` 返回类型是 `String`（只取 `.call().content()`），丢弃了 `ChatResponse` 中的 `usage` 元数据：

```java

private String callWithRoles(ChatClient client, String systemPrompt, String userPrompt) {
    return client.prompt()
        .system(systemPrompt)
        .user(userPrompt)
        .call()
        .content();  // ← 只取 content，丢掉 ChatResponse
}
```

### 4.3 配置 OpenTelemetry 自动装备

**新增文件**: `src/main/java/com/example/deepresearch/common/config/ObservabilityConfig.java`

```java
package com.example.deepresearch.common.config;

import io.micrometer.core.instrument.MeterRegistry;
import io.micrometer.core.aop.TimedAspect;
import io.micrometer.observation.ObservationRegistry;
import io.micrometer.observation.aop.ObservedAspect;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

/**
 * 可观测性配置 — 启用 @Timed 和 @Observed AOP 切面.
 * <p>
 * Spring Boot Actuator 自动配置处理了大部分基础设施（MeterRegistry、
 * ObservationRegistry、Tracing），这里只需显式注册 AOP 切面 Bean，
 * 使 {@code @Timed} 和 {@code @Observed} 注解生效。
 * </p>
 *
 * <h3>使用方式</h3>
 * <pre>
 * {@code @Timed(value = "deepresearch.workflow.duration", extraTags = {"node", "plan"})}
 * public PlanResult plan(String query) { ... }
 * </pre>
 */
@Configuration(proxyBeanMethods = false)
public class ObservabilityConfig {

    /**
     * 启用 @Timed 注解的 AOP 切面.
     * 允许在任意 Spring Bean 方法上使用 @Timed 自动记录耗时和调用次数。
     */
    @Bean
    public TimedAspect timedAspect(MeterRegistry meterRegistry) {
        return new TimedAspect(meterRegistry);
    }

    /**
     * 启用 @Observed 注解的 AOP 切面.
     * 允许在任意 Spring Bean 方法上使用 @Observed 自动创建 Observation（含 Span）。
     */
    @Bean
    public ObservedAspect observedAspect(ObservationRegistry observationRegistry) {
        return new ObservedAspect(observationRegistry);
    }
}
```

### 5.2 创建工作流 Tracing 工具类

**新增文件**: `src/main/java/com/example/deepresearch/common/observability/WorkflowTracingHelper.java`

```java
package com.example.deepresearch.common.observability;

import io.micrometer.observation.Observation;
import io.micrometer.observation.ObservationRegistry;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.stereotype.Component;

import java.util.function.Supplier;

/**
 * 工作流 Tracing 助手 — 为每个工作流节点创建 Observation（含 Span）.
 * <p>
 * 使用 Micrometer Observation API 创建 Span，同时自动集成到
 * OpenTelemetry 导出链中。每个 Observation 自动生成 metrics（Timer + Counter）
 * 和 tracing span。
 * </p>
 *
 * <h3>使用方式</h3>
 * <pre>
 * {@code
 * PlanResult result = tracingHelper.observe("workflow.plan",
 *     Map.of("sessionId", sessionId),
 *     () -> planner.plan(query, memoryContext));
 * }
 * </pre>
 */
@Component
public class WorkflowTracingHelper {

    private static final Logger log = LoggerFactory.getLogger(WorkflowTracingHelper.class);

    private final ObservationRegistry observationRegistry;

    public WorkflowTracingHelper(ObservationRegistry observationRegistry) {
        this.observationRegistry = observationRegistry;
    }

    /**
     * 在 Observation 上下文中执行操作.
     *
     * @param name   Observation 名称（如 "workflow.plan"），自动成为 Span 名称
     * @param tags   自定义标签（key-value），会附加到 Span 属性
     * @param action 要执行的操作
     * @return 操作结果
     */
    public <T> T observe(String name, java.util.Map<String, String> tags,
                         Supplier<T> action) {
        Observation observation = Observation.createNotStarted(name, observationRegistry)
            .lowCardinalityKeyValue("application", "deep-research-agent")
            .contextualName(name);

        // 添加自定义标签
        if (tags != null) {
            for (var entry : tags.entrySet()) {
                observation = observation.lowCardinalityKeyValue(entry.getKey(), entry.getValue());
            }
        }

        return observation.observe(() -> {
            log.debug("[Tracing] {} started", name);
            try {
                T result = action.get();
                log.debug("[Tracing] {} completed", name);
                return result;
            } catch (Exception e) {
                observation.error(e);
                log.error("[Tracing] {} failed: {}", name, e.getMessage());
                throw e;
            }
        });
    }

    /**
     * 创建 Observation 但不自动执行（用于与 CompletableFuture 结合）.
     * 调用方需要手动调用 {@code observation.observe(action)}.
     */
    public Observation createObservation(String name, java.util.Map<String, String> tags) {
        Observation observation = Observation.createNotStarted(name, observationRegistry)
            .lowCardinalityKeyValue("application", "deep-research-agent")
            .contextualName(name);

        if (tags != null) {
            for (var entry : tags.entrySet()) {
                observation = observation.lowCardinalityKeyValue(entry.getKey(), entry.getValue());
            }
        }
        return observation;
    }
}
```

## 6. Phase 3: Token 监控集成 — TokenTrackingAdvisor

这是当前最关键的缺口。`TokenUsageTracker` 实现完美但未被调用。

### 6.1 核心思路

当前 `ModelFallbackService.callWithRoles()` 的问题：

```java
// ❌ 当前 — 丢失 ChatResponse metadata
return client.prompt().system(sp).user(up).call().content();

// ✅ 应该 — 捕获 ChatResponse 以提取 Usage
ChatResponse response = client.prompt().system(sp).user(up).call().chatResponse();
Usage usage = response.getMetadata().getUsage();
tokenTracker.track(agentName, modelTier, 
    usage.getPromptTokens(), usage.getGenerationTokens(),
    System.currentTimeMillis() - start, sessionId);
return response.getResult().getOutput().getText();
```

但是 `ModelFallbackService` 是给 Planner/Writer 这些使用 Pro→Flash 降级的 Agent 用的，其他 5 个 Flash Agent 直接调用 `ChatClient.call()`，没有经过 `ModelFallbackService`。

**更好的方案**: 创建一个 **`TokenTrackingAdvisor`**（类似 `PiiMaskingAdvisor`），注册到**所有** ChatClient 的 `defaultAdvisors()` 中，**自动拦截每一次 LLM 调用**，零侵入。

### 6.2 创建 TokenTrackingAdvisor

**新增文件**: `src/main/java/com/example/deepresearch/common/observability/TokenTrackingAdvisor.java`

```java
package com.example.deepresearch.common.observability;

import io.micrometer.core.instrument.MeterRegistry;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.ai.chat.client.ChatClientRequest;
import org.springframework.ai.chat.client.ChatClientResponse;
import org.springframework.ai.chat.client.advisor.api.BaseAdvisor;
import org.springframework.ai.chat.client.advisor.api.AdvisorChain;
import org.springframework.ai.chat.metadata.Usage;
import org.springframework.ai.chat.model.ChatResponse;
import org.springframework.core.Ordered;
import org.springframework.stereotype.Component;

/**
 * Token 追踪 Advisor — 透明拦截所有 ChatClient 调用，记录 Token 用量.
 * <p>
 * 通过 Spring AI 的 {@link BaseAdvisor} 机制，在每次 ChatClient 调用前后
 * 自动记录 LLM 调用的 Token 消耗、延迟和估算成本。
 * </p>
 *
 * <h3>工作原理</h3>
 * <ol>
 *   <li>{@code before()}: 记录调用开始时间（存入 AdvisorContext）</li>
 *   <li>{@code after()}: 从 ChatResponse 提取 Usage，计算耗时，写入 MeterRegistry</li>
 * </ol>
 *
 * <h3>Agent 名称识别</h3>
 * <p>
 * 从 ChatClientRequest 的 advisorParams 中读取 "agent" 和 "tier" 参数。
 * 调用方在构建 prompt 时通过 {@code .advisors(a -> a.param("agent", "planner").param("tier", "pro"))}
 * 传入。如果未指定，默认使用 "unknown" / "unknown"。
 * </p>
 *
 * <h3>优先级</h3>
 * {@link Ordered#LOWEST_PRECEDENCE + 100} — 在 PiiMaskingAdvisor 之后执行，
 * 确保 PII 脱敏先完成，但仍在大多数 Advisor 之后。
 */
@Component
public class TokenTrackingAdvisor implements BaseAdvisor {

    private static final Logger log = LoggerFactory.getLogger(TokenTrackingAdvisor.class);

    /** AdvisorContext 中存储开始时间的 key */
    static final String START_TIME_KEY = "token_tracker.start_time";

    private final TokenUsageTracker tokenTracker;

    public TokenTrackingAdvisor(TokenUsageTracker tokenTracker) {
        this.tokenTracker = tokenTracker;
    }

    @Override
    public String getName() {
        return "TokenTrackingAdvisor";
    }

    @Override
    public int getOrder() {
        // 在 PiiMaskingAdvisor (HIGHEST_PRECEDENCE) 之后执行
        return Ordered.LOWEST_PRECEDENCE + 100;
    }

    @Override
    public ChatClientRequest before(ChatClientRequest request, AdvisorChain chain) {
        // 记录调用开始时间
        request = request.mutate()
            .contextPut(START_TIME_KEY, System.currentTimeMillis())
            .build();
        return request;
    }

    @Override
    public ChatClientResponse after(ChatClientResponse response, AdvisorChain chain) {
        ChatResponse chatResponse = response.chatResponse();
        if (chatResponse == null) {
            return response;
        }

        try {
            // 提取 Usage 元数据
            Usage usage = chatResponse.getMetadata().getUsage();
            if (usage != null) {
                // 从 Advisor Context 中恢复开始时间
                Long startTime = response.context().get(START_TIME_KEY);
                long latencyMs = (startTime != null)
                    ? System.currentTimeMillis() - startTime
                    : 0;

                // 提取 Agent 和模型信息
                // 优先从 advisor params 读取，fallback 到 ChatResponse metadata
                String agentName = getStringParam(response, "agent", "unknown");
                String modelTier = getStringParam(response, "tier", "unknown");

                // 追踪 Token 用量
                tokenTracker.track(
                    agentName,
                    modelTier,
                    usage.getPromptTokens(),
                    usage.getGenerationTokens(),
                    latencyMs,
                    getStringParam(response, "sessionId", "")
                );

                log.debug("[TokenTracker] {} ({}): prompt={}, gen={}, latency={}ms",
                    agentName, modelTier,
                    usage.getPromptTokens(), usage.getGenerationTokens(), latencyMs);
            }
        } catch (Exception e) {
            // Token 追踪失败不应影响业务流程
            log.warn("[TokenTracker] 追踪失败（非致命）: {}", e.getMessage());
        }

        return response;
    }

    /**
     * 从 Advisor Context 或 ChatResponse metadata 中提取字符串参数.
     */
    private String getStringParam(ChatClientResponse response, String key, String defaultValue) {
        // 先查 Advisor Context
        Object value = response.context().get(key);
        if (value != null) {
            return value.toString();
        }
        // Fallback 到 ChatResponse metadata
        if (response.chatResponse().getMetadata() != null) {
            Object metaValue = response.chatResponse().getMetadata().get(key);
            if (metaValue != null) {
                return metaValue.toString();
            }
        }
        return defaultValue;
    }
}
```

**修改 `ModelFallbackService.callWithRoles()` 返回 ChatResponse**：

这里需要做一个重要决策。当前 `callWithRoles()` 只返回 `String`（content）。要提取 Usage，需要改为返回完整响应。但改动签名会影响所有 Agent。

**推荐方案**: 增加一个带 advisor params 的重载，agent 调用时通过 `advisors()` 传入参数：

```java
// ModelFallbackService.java — 修改 callWithRoles
private String callWithRoles(ChatClient client, String systemPrompt,
                              String userPrompt, String agentName, String modelTier) {
    long startTime = System.currentTimeMillis();
    
    ChatClient.CallResponseSpec callSpec = client.prompt()
        .advisors(a -> a
            .param("agent", agentName)
            .param("tier", modelTier))
        .system(systemPrompt)
        .user(userPrompt)
        .call();
    
    // ChatClientResponse 包含 ChatResponse（含 Usage）
    ChatResponse chatResponse = callSpec.chatResponse();
    
    // TokenTrackingAdvisor 已在 after() 中自动追踪
    // 这里只需正常返回 content
    
    return chatResponse.getResult().getOutput().getText();
}
```

## 9. 最佳实践建议

### 9.1 指标命名规范

遵循 Prometheus 命名最佳实践：

```
deepresearch_{domain}_{metric}_{unit}

示例:
  deepresearch_llm_calls_total                    # Counter
  deepresearch_llm_latency_seconds                # Timer (Histogram)
  deepresearch_workflow_duration_seconds          # Timer
  deepresearch_cache_access_total                 # Counter
```

> **规则**:
> - 使用 `_total` 后缀命名 Counter
> - 使用基本单位（seconds, bytes），不用 milliseconds/megabytes
> - 标签值使用小写下划线（snake_case）

### 9.2 高基数标签控制

❌ **避免**:
```java
Counter.builder("deepresearch.llm.calls")
    .tag("query", query)           // ← 每个查询不同，标签爆炸！
    .tag("prompt", promptText)     // ← 同上
```

✅ **推荐**:
```java
Counter.builder("deepresearch.llm.calls")
    .tag("agent", "planner")       // ← 有限基数（7 个 Agent）
    .tag("tier", "pro")            // ← 有限基数（2 个 tier）
    .tag("status", "success")      // ← 有限基数（~3 个 status）
```

> **规则**: 标签值的基数应小于 100。超过此值应考虑使用 Exemplar 或日志记录。

### 9.3 采样策略

| 环境 | Tracing 采样率 | 理由 |
|:---|:---|:---|
| 开发 | 100% (`1.0`) | 调试需要完整 trace |
| 测试 | 100% (`1.0`) | 发现性能瓶颈 |
| 生产 | 10-25% (`0.1-0.25`) | 平衡可观测性与性能开销 |
| 高流量生产 | 1-5% (`0.01-0.05`) | 最小化开销 |

### 9.4 日志结构化

当前日志使用模式字符串，建议升级为 JSON 结构化日志（生产环境）：

```yaml
# application-prod.yml
logging:
  pattern:
    console: ""  # 生产环境禁用控制台
  file:
    name: logs/research.json
  logback:
    rollingpolicy:
      file-name-pattern: logs/research.%d{yyyy-MM-dd}.%i.json.gz
  # 使用 logstash-logback-encoder 实现 JSON 格式
```

**推荐依赖**:
```xml
<dependency>
    <groupId>net.logstash.logback</groupId>
    <artifactId>logstash-logback-encoder</artifactId>
    <version>7.4</version>
</dependency>
```

### 9.5 成本归属标签

TokenTrackingAdvisor 的 `agent` 和 `tier` 标签是成本归属的关键。确保在 Grafana 中能按此维度拆分成本：

```promql
# 按 Agent 拆分的每小时成本
sum(rate(deepresearch_llm_cost_estimated_total[1h])) by (agent) * 3600

# 按 Tier 拆分的每日成本
sum(increase(deepresearch_llm_cost_estimated_total[24h])) by (tier) / 1000000
```

### 9.6 安全指标监控

PII 脱敏和注入检测指标应设置高优先级告警：

| 指标 | 告警条件 | 严重级别 |
|:---|:---|:---|
| `deepresearch_security_injection_detected_total` | rate > 0 | 🔴 Critical |
| `deepresearch_llm_fallback_total` | rate > 0.05/s | 🟠 Warning |
| `deepresearch_security_pii_masked_total` | — | 🟢 Info（仅监控趋势） |

### 9.7 性能开销预算

| 组件 | 开销 | 说明 |
|:---|:---|:---|
| Micrometer Metrics | ~0.1% CPU | 内存内聚合，极低开销 |
| Prometheus Scrape | ~1-5ms/抓取 | 依赖指标数量和标签基数 |
| Tracing (10% 采样) | ~0.5% CPU | 创建 span 有一定开销 |
| OTLP Export | ~1-2ms/批次 | 异步批量发送，不阻塞请求线程 |
| JSON 日志 | ~5-10% CPU | 比模式字符串开销大，仅生产环境推荐 |
